  EXPERIMENT 6 — Scheduler Recommendation by Task Size
════════════════════════════════════════════════════════════════════════

  Based on timing breakpoints and quality analysis above.

  ┌─────────────────────┬──────────────────────────────────────────────────────┐
  │  Task-size range     │  Recommended scheduler(s)                            │
  ├─────────────────────┼──────────────────────────────────────────────────────┤
  │  n ≤ 12              │  MILP  (provably optimal; reliably <5s)              │
  │  12 < n ≤ 20         │  MILP + 20s time limit (suboptimal incumbent ok)     │
  │  20 < n ≤ 100        │  HEFT or MinMin  (within ~5% of MILP optimum)        │
  │  100 < n ≤ 500       │  HEFT or Greedy  (sub-millisecond)                   │
  │  n > 500             │  Metaheuristic (GA/SA/ACO) or Dagger.jl native       │
  └─────────────────────┴──────────────────────────────────────────────────────┘

  Empirical evidence for the n=12 threshold:
  • n=10: solves in 1–9s (instance-dependent), gap closes to 0%
  • n=12: solves in 2–5s, reliable across seeds
  • n=15: B&B explores 11,000+ nodes, gap stuck at 42% after 35s
  • n=20: never closes within a practical budget
  Root cause: binary o[T,T,K] variables grow as O(n²·|K|).

  Why HEFT is the default heuristic choice:
  • Prioritises tasks by upward rank (critical-path proxy)
  • O(n·|K|·log n) — fast even at n=500
  • Consistently within 1–5% of MILP optimum on random DAGs
  • Well-studied: used as the standard baseline in scheduling literature

  Next steps for n > 500:
  • Genetic Algorithm with HEFT-seeded initial population
  • Simulated Annealing on device assignment permutations
  • Ant Colony Optimisation along DAG critical paths
  • Dagger.jl native EagerScheduler / ThriftScheduler integration
